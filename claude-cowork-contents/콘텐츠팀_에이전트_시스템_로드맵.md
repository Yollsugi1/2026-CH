### Layer 1 — 공유 스킬 (이미 존재)

여러 에이전트가 공통으로 호출하는 지식/도구 모듈.

| 스킬 | 호출하는 에이전트 |
|---|---|
| `content-planner` | 시리즈 기획 에이전트들 (A-B-C 프레임워크 적용 시) |
| `content-frameworks` | 시리즈 기획 에이전트들 (프레임워크 레퍼런스 조회 시) |
| `ch-mag-writer` | 캐러셀 콘텐츠 기획 시 직접 호출 (스킬 유지) |
| `ch-mag-short-caption` | `copywrite-agent`가 참조 (기존 톤 레퍼런스) |
| `bwasseo-script` | 봤어? 기획 시 직접 호출 (스킬 유지) |

### Layer 2 — 에이전트

두 종류로 구분:

**A. 시리즈 기획 에이전트** — 각 시리즈의 기획~사전제작 워크플로우
- `hoodies-planner`, `playbook-planner`, `shortform-planner`
- (추후) `false9-planner`, `hood-residents-planner`, `shirtless-planner`

**B. 기능 에이전트** — 시리즈 횡단적 기능
- `copywrite-agent` — 콘텐츠 업로드 시 제목/캡션 작성

### Layer 3 — Contents PM

`contents-pm` — 주간 콘텐츠 플래닝을 진행하고, 시리즈별 에이전트를 디스패치하며, 산출물을 수집/리뷰하는 PM 에이전트.

## 3. 에이전트 구분 체계

### 구분 원칙

```
콘텐츠 제작 파이프라인:

[기획]           [제작]         [업로드]
  │                │               │
  ▼                ▼               ▼
시리즈 기획     (사람이 촬영/    copywrite-agent
에이전트          편집)          (제목, 캡션, 해시태그)
  │                               │
  ├─ hoodies-planner              ├─ YouTube 제목/설명
  ├─ playbook-planner             ├─ Instagram 캡션
  ├─ shortform-planner            └─ 해시태그 세트
  └─ (추후 3종)

[캐러셀/봤어? = 스킬로 유지]
  ├─ ch-mag-writer (캐러셀 에디토리얼)
  └─ bwasseo-script (봤어? 스크립트)

[주간 운영]
  └─ contents-pm (플래닝 + 디스패치 + 리뷰)
```

### 에이전트 vs 스킬 기준

| | 에이전트 (Agent) | 스킬 (Skill) |
|---|---|---|
| 판단/적응 필요 | O | X |
| 다단계 추론 | O | X |
| 사용자 피드백 루프 | O | X |
| 병렬 리서치 | O | X |
| 해당하는 것 | hoodies, playbook, shortform, copywrite, contents-pm | ch-mag-writer, bwasseo-script, ch-mag-short-caption |

**ch-mag-writer를 스킬로 유지하는 이유:**
- 소재 접수 → 앵글 기획 → Draft → 피드백 → Instagram 변환이라는 고정 파이프라인
- 사용자와 1:1 대화형으로 진행하는 특성 (에이전트로 자율 실행하기보단 스킬로 가이드)

**bwasseo-script를 스킬로 유지하는 이유:**
- 단일 소스 → 단일 산출물 패턴
- 리서치 → 스크립트 → 공유본의 결정적 파이프라인

## 4. 시리즈 기획 에이전트 상세 설계

### 4-1. Hoodies (후디스) — `hoodies-planner`

**복잡도:** ★★★★★ (가장 복잡)
**특성:** Person-centric, 다세트 구조, 촬영 가이드 필요
**모델:** sonnet

**파이프라인:**
```
Phase 1: 출연자 프로필 수집
  └ 이름, 본업, 운동, 배경, 삶의 맥락
Phase 2: 캐릭터 핵심 읽기
  └ "이 사람은 ___으로 사는 사람이다" 한 줄 정의
  └ 키워드 3~5개 도출
Phase 3: 세트 구조 설계 (사용자 피드백 필수)
  └ 3EP 1세트 원칙
  └ 시간순/테마순 세트 배치
  └ 세트 간 감정 아크 (크레센도)
Phase 4: 세트별 기획
  └ 컨셉, EP별 스토리라인
  └ 공간 플로우, 시각적 키, 감정 아크
  └ 촬영 활용 정보 (장소, 소품, 인물)
Phase 5: 촬영 가이드 작성
  └ 대화 운영 원칙 ("질문하지 않는다. 대화한다.")
  └ 대화 갈래 지도 (갈래 A/B/C + 더 갈 수 있는 방향)
  └ EP별 에피소드 구조 (HOOK → 만남 → 세계 → 깊은 대화 → 마무리)
```

**인풋:** 출연자 정보 (이름, 배경, 운동)
**산출물:**
1. `hoodies-{출연자명}-plan.md` — 전체 세트 기획안
2. `hoodies-{출연자명}-shooting-guide.md` — 촬영 가이드 + 대화 갈래 지도
3. 세트별 EP 디테일 (공간 플로우, 시각적 키, 감정 아크)

**레퍼런스:**
- `hoodies/hoodies-series-master-plan.md` → 시리즈 DNA, 에피소드 구조 템플릿
- `hoodies/ahn-sunghee/` → 세트/EP 기획 톤과 깊이 레퍼런스

### 4-2. Playbook (플레이북) — `playbook-planner` ✅ 완성

**복잡도:** ★★★★☆
**특성:** Topic-centric, 소주제 4개 병렬 리서치, 호스트 스크립트
**모델:** sonnet

**파이프라인:** (이미 구현됨)
```
Phase 1: 인풋 수집 (대주제, 게스트, EP번호)
Phase 2: 소주제 구조 설계 (사용자 피드백)
Phase 3: 4개 병렬 리서치 에이전트
Phase 4: 기획안 컴파일
Phase 5: 호스트 스크립트 작성
```

**산출물:**
1. `Playbook_EP{N}_{주제명}_기획안.md`
2. `Playbook_EP{N}_호스트_스크립트.md`

### 4-3. 숏폼 시리즈 — `shortform-planner`

**복잡도:** ★★☆☆☆
**특성:** 장소/테마 기반 숏폼 기획
**대상 시리즈:** 폭력적인 식단, Pit Stop (현재) / Shirtless (추후 추가)
**모델:** sonnet

**파이프라인:**
```
Phase 1: 소스 수집
  └ 시리즈 파라미터 확인 (폭력적인 식단 / Pit Stop)
  └ 장소명/테마, 위치, 키워드
Phase 2: 리서치
  └ WebSearch로 장소/테마 정보 수집
  └ 시리즈 톤에 맞는 앵글 도출
Phase 3: 스크립트 작성
  └ 시리즈별 구조 분기:
    - 폭력적인 식단: 훅(3초) → 장소(15초) → 음식/경험(30초) → CTA(10초)
    - Pit Stop: 훅(3초) → 공간 무드(15초) → 큐레이션(30초) → CTA(10초)
  └ 비주얼 디렉션 포함
Phase 4: 촬영 체크리스트
  └ 필요 장비, 주문/준비 리스트
```

**시리즈별 톤 분기:**
| | 폭력적인 식단 | Pit Stop |
|---|---|---|
| 톤 | 에너제틱, 강렬 | 차분, 큐레이션 |
| 앵글 | 운동 후 식사 | 운동인의 피트스탑 |
| 비주얼 | 다이나믹 | 시네마틱/무드 |

**인풋:** 시리즈명 + 장소명 or 테마
**산출물:** `{시리즈명}_{장소명}_스크립트.md`

**Shirtless 추가 시:** 톤 분기 테이블에 Shirtless 행 추가 + Phase 3에 스타일링 구조 분기 추가

## 5. 기능 에이전트 상세 설계

### 5-1. Copywrite Agent — `copywrite-agent`

**역할:** 콘텐츠 업로드 시 인스타그램/유튜브에 올릴 제목과 캡션을 작성
**복잡도:** ★★★☆☆
**모델:** sonnet

**파이프라인:**
```
Phase 1: 콘텐츠 파악
  └ 시리즈명, 에피소드 정보
  └ 기획안/스크립트 읽기 (시리즈 기획 에이전트의 산출물)
  └ 플랫폼 확인 (YouTube / Instagram / 둘 다)

Phase 2: 플랫폼별 카피 작성
  └ YouTube:
    - 제목 (후킹 + 핵심 키워드, 60자 이내)
    - 설명문 (요약 + 타임스탬프 + CTA + 해시태그)
  └ Instagram:
    - 캡션 (시리즈 톤에 맞는 1~3문장 카피 + 본문)
    - 해시태그 세트 (브랜드 + 시리즈 + 주제 관련)

Phase 3: 시리즈별 톤 적용
  └ Hoodies: 진정성, 깊이, 인물 중심
  └ Playbook: 지적 호기심, 대화체
  └ 폭력적인 식단: 강렬, 에너제틱
  └ Pit Stop: 차분, 무드
  └ 봤어?: 궁금증 유발, 빠른 톤
  └ (시리즈별 톤 가이드 참조)

Phase 4: 산출물 정리
  └ 플랫폼별 복붙 가능한 형태로 정리
```

**인풋:** 시리즈명 + 에피소드 기획안/스크립트 경로 + 플랫폼
**산출물:** `{시리즈}_{에피소드}_upload_copy.md`

**기존 `ch-mag-short-caption` 스킬과의 관계:**
- `ch-mag-short-caption`은 포토덤프/단편 이미지용 짧은 카피에 특화
- `copywrite-agent`는 영상 콘텐츠 업로드용 제목+캡션에 특화
- 둘은 공존 — 용도가 다름. `copywrite-agent`가 `ch-mag-short-caption`의 톤을 참고할 수 있음

### 5-2. Contents PM — `contents-pm`

**역할:** 주간 콘텐츠 플래닝을 진행하고, 시리즈별 에이전트를 병렬 디스패치하며, 산출물을 수집/리뷰
**복잡도:** ★★★★☆
**모델:** sonnet

**PM의 3가지 역할:**

**A. 주간 플래닝 진행**
```
Phase 1: 주간 플랜 수신
  └ 사용자가 이번 주 제작할 콘텐츠 목록 제공
  └ 또는 PM이 스케줄/마스터플랜 기반으로 이번 주 할 일 제안

Phase 2: 플랜 정리 + 확인
  └ 시리즈별 태스크 분리
  └ 우선순위 설정
  └ 사용자 컨펌
```

**B. 에이전트 디스패치**
```
Phase 3: 병렬 에이전트 디스패치 (TeamCreate API)
  └ 시리즈 기획 에이전트들 병렬 런칭
  └ 봤어?/캐러셀은 스킬 직접 호출
  └ 선택적 디스패치 (플랜에 포함된 시리즈만)

Phase 4: 진행 상태 추적
  └ TaskCreate/TaskUpdate API로 에이전트별 상태 관리
  └ 사용자 피드백이 필요한 에이전트 알림 (Playbook 소주제, Hoodies 세트 구조 등)
```

**C. 산출물 수집 + 리뷰**
```
Phase 5: 산출물 수집
  └ 각 에이전트의 완료 파일 수집
  └ copywrite-agent 연동 (기획 완료된 콘텐츠의 업로드 카피 생성)

Phase 6: 통합 리뷰 + 리포트
  └ 시리즈 간 주제 겹침 체크
  └ 톤 일관성 확인
  └ 이번 주 기획 완료 현황 리포트
```

**인풋 형식:**

```markdown
## 이번 주 콘텐츠 플랜

### Playbook
- 주제: [대주제]
- 게스트: [이름] ([배경])
- EP 번호: [N]

### 봤어?
- 소스 링크: [URL]

### 폭력적인 식단
- 장소: [장소명]

### Pit Stop
- 카페: [카페명]

### Hoodies
- 출연자: [이름]
- 이번 주 작업: [세트 N 기획 / 촬영 가이드]

### 캐러셀
- 소재: [주제/링크]
```

**산출물:** 주간 기획 리포트 + 시리즈별 기획 파일 + 업로드 카피

## 6. 에이전트 구현 우선순위

### Tier 1 — 즉시 구현 (레퍼런스 충분)

| 순서 | 에이전트 | 이유 |
|---|---|---|
| 1 | `hoodies-planner` | 안성희 기획 1,360줄의 풍부한 레퍼런스. 패턴 명확 |
| 2 | `shortform-planner` | Pit Stop 호스트가이드 + 봤어? 스킬 구조 참고. 식단+Pit Stop 2시리즈 |
| 3 | `copywrite-agent` | 기존 ch-mag-short-caption 톤 참조. 모든 시리즈에 즉시 활용 가능 |

### Tier 2 — Tier 1 완성 후

| 순서 | 에이전트 | 이유 |
|---|---|---|
| 4 | `contents-pm` | Layer 2 에이전트가 3개 이상 완성된 후 PM 구축. 디스패치 대상이 있어야 의미 |

### Tier 3 — 레퍼런스 축적 후 (시리즈 시작 후)

| 순서 | 에이전트 | 이유 |
|---|---|---|
| 5 | `false9-planner` | 첫 1~2편 기획 완료 후 패턴 추출 |
| 6 | `hood-residents-planner` | 2~3편 기획 완료 후 |
| 7 | `shirtless-planner` 또는 `shortform-planner` 확장 | 시리즈 론칭 후 |

### 구현 마일스톤

```
Session A: hoodies-planner 에이전트 작성
  └ 마스터플랜 + 안성희 기획을 레퍼런스로 주입
  └ 5단계 파이프라인 구현
  └ 테스트: 새 출연자(고예원 or 백주현) 기획 시뮬레이션

Session B: shortform-planner 에이전트 작성
  └ 폭력적인 식단 + Pit Stop 2시리즈 통합
  └ 시리즈 파라미터에 따른 톤/구조 분기
  └ 테스트: 실제 장소로 스크립트 생성

Session C: copywrite-agent 에이전트 작성
  └ 플랫폼별(YouTube/Instagram) 카피 생성
  └ 시리즈별 톤 가이드 정의
  └ ch-mag-short-caption 참조하되 영상 콘텐츠에 최적화
  └ 테스트: 기존 에피소드로 카피 생성하여 실제 업로드본과 비교

Session D: contents-pm 에이전트 작성
  └ 주간 플랜 파싱 + 에이전트 디스패치
  └ TeamCreate API로 병렬 실행
  └ 산출물 수집 + copywrite-agent 연동 + 리포트
  └ 테스트: 실제 주간 플래닝으로 전체 파이프라인 시뮬레이션

(Tier 3 — 시리즈 경험 축적 후 별도 세션)
```

### 7-1. 공통 YAML 프론트매터

### 7-2. 공통 파이프라인 패턴

모든 시리즈 기획 에이전트는 다음 구조를 공유:

```
Phase 1: 인풋 수집/파악
Phase 2: 구조 설계 (사용자 피드백 포인트)
Phase 3: 리서치 (WebSearch/WebFetch)
Phase 4: 산출물 작성
Phase 5: 메타 정보 정리
```

시리즈별 차이는 각 Phase의 깊이와 산출물 형식에 있음.

## 10. 구현 시 주의사항

### 10-1. 에이전트 크기 제한

- 시스템 프롬프트가 너무 길면 컨텍스트 소모 → 핵심만 프롬프트에, 디테일은 `references/` 폴더에 분리
- Hoodies처럼 레퍼런스가 방대한 경우: 에이전트가 런타임에 `read`로 레퍼런스 로드

### 10-2. contents-pm 점진적 구축

처음부터 완전한 PM을 만들지 말 것.

```
v0.1: 2~3개 에이전트를 수동으로 병렬 디스패치 (Agent 도구 직접 호출)
v0.2: 간단한 PM (플랜 파싱 + 디스패치만)
v0.3: 산출물 수집 + copywrite-agent 연동
v1.0: 완전 자동화 (플랜 입력 → 기획 + 카피 + 리포트)
```

### 10-3. 테스트 전략

각 에이전트 완성 후:
1. **실제 소재로 테스트** — 가상 소재가 아닌 실제 기획 소재로 실행
2. **산출물 품질 검증** — 기존 수동 기획 결과물과 비교
3. **사용자 피드백 루프 동작 확인** — 중간 확인이 필요한 단계에서 정상 작동하는지

### 10-4. /save, /resume는 커맨드 유지

세션 백업 + git sync는 판단이 필요 없는 결정적 워크플로우.
현재 `.claude/commands/save.md`, `.claude/commands/resume.md`로 충분.
에이전트 전환 불필요.

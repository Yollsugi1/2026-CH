## 3. 시리즈별 에이전트 상세 설계

### 3-1. Hoodies (후디스) — `hoodies-planner`

**복잡도:** ★★★★★ (가장 복잡)
**특성:** Person-centric, 다세트 구조, 촬영 가이드 필요
**모델:** sonnet

**파이프라인:**
```
Phase 1: 출연자 프로필 수집
  └ 이름, 본업, 운동, 배경, 삶의 맥락
Phase 2: 캐릭터 핵심 읽기
  └ "이 사람은 ___으로 사는 사람이다" 한 줄 정의
  └ 키워드 3~5개 도출
Phase 3: 세트 구조 설계 (사용자 피드백 필수)
  └ 3EP 1세트 원칙
  └ 시간순/테마순 세트 배치
  └ 세트 간 감정 아크 (크레센도)
Phase 4: 세트별 기획
  └ 컨셉, EP별 스토리라인
  └ 공간 플로우, 시각적 키, 감정 아크
  └ 촬영 활용 정보 (장소, 소품, 인물)
Phase 5: 촬영 가이드 작성
  └ 대화 운영 원칙 ("질문하지 않는다. 대화한다.")
  └ 대화 갈래 지도 (갈래 A/B/C + 더 갈 수 있는 방향)
  └ EP별 에피소드 구조 (HOOK → 만남 → 세계 → 깊은 대화 → 마무리)
```

**인풋:** 출연자 정보 (이름, 배경, 운동)
**산출물:**
1. `hoodies-{출연자명}-plan.md` — 전체 세트 기획안
2. `hoodies-{출연자명}-shooting-guide.md` — 촬영 가이드 + 대화 갈래 지도
3. 세트별 EP 디테일 (공간 플로우, 시각적 키, 감정 아크)

**레퍼런스로 사용:**
- `hoodies/hoodies-series-master-plan.md` → 시리즈 DNA, 에피소드 구조 템플릿
- `hoodies/ahn-sunghee/` → 세트/EP 기획 톤과 깊이 레퍼런스

### 3-2. Playbook (플레이북) — `playbook-planner` ✅ 완성

**복잡도:** ★★★★☆
**특성:** Topic-centric, 소주제 4개 병렬 리서치, 호스트 스크립트
**모델:** sonnet

**파이프라인:** (이미 구현됨)
```
Phase 1: 인풋 수집 (대주제, 게스트, EP번호)
Phase 2: 소주제 구조 설계 (사용자 피드백)
Phase 3: 4개 병렬 리서치 에이전트
Phase 4: 기획안 컴파일
Phase 5: 호스트 스크립트 작성
```

**산출물:**
1. `Playbook_EP{N}_{주제명}_기획안.md`
2. `Playbook_EP{N}_호스트_스크립트.md`

### 3-3. 봤어? — 스킬로 충분 (에이전트 불필요)

**복잡도:** ★★☆☆☆
**특성:** 링크/주제 → 리서치 → 4단 스크립트 + Manychat 공유본

`bwasseo-script` 스킬이 이미 전체 워크플로우를 커버. 단일 소스 → 단일 산출물 패턴이라 에이전트 수준의 판단이 불필요.

**오케스트레이터 연동:** 스킬을 직접 호출하거나, 얇은 래퍼 에이전트로 감싸서 디스패치.

### 3-4. 폭력적인 식단 — `shortform-planner` (공유)

**복잡도:** ★★☆☆☆
**특성:** 장소 기반 숏폼, 맛집 큐레이션

**파이프라인:**
```
Phase 1: 소스 수집
  └ 장소명, 위치, 메뉴 키워드
Phase 2: 리서치
  └ WebSearch로 장소 정보, 리뷰, 특징 수집
  └ 운동/스포츠 라이프스타일 연결 앵글 도출
Phase 3: 스크립트 작성
  └ 훅 (3초) → 장소 소개 (15초) → 음식/경험 (30초) → 마무리 CTA (10초)
  └ 비주얼 디렉션 포함
Phase 4: 메타 정보
  └ 촬영 체크리스트, 필요 장비, 음식 주문 리스트
```

**인풋:** 장소명 or 테마
**산출물:** `폭력적인식단_{장소명}_스크립트.md`

### 3-5. Pit Stop — `shortform-planner` (공유) 또는 독립 에이전트

**복잡도:** ★★☆☆☆
**특성:** 카페 큐레이션 숏폼

폭력적인 식단과 구조가 유사 (장소 기반 숏폼). `shortform-planner`를 공유하되 시리즈별 톤/구조 분기를 파라미터로 처리.

**차이점:**
- 톤: 폭력적인 식단 = 에너제틱/강렬, Pit Stop = 차분/큐레이션
- 앵글: 폭력적인 식단 = 운동 후 식사, Pit Stop = 운동인의 피트스탑
- 비주얼: 폭력적인 식단 = 다이나믹, Pit Stop = 시네마틱/무드

**기존 레퍼런스:** `PitStop_기획서_호스트가이드.md` → 호스트 가이드 + 시리즈 톤 참고

---

### 3-6. Shirtless — `shortform-planner` (공유) 또는 독립 에이전트

**복잡도:** ★★☆☆☆
**특성:** 스타일링 숏폼

**파이프라인:**
```
Phase 1: 소스 수집
  └ 스타일링 테마, 아이템, 출연자
Phase 2: 리서치
  └ 트렌드 검색, 브랜드/아이템 정보
  └ 스포츠 라이프스타일 연결점
Phase 3: 스크립트 작성
  └ 훅 → 아이템 소개 → 스타일링 팁 → 마무리
  └ 비주얼 디렉션 (앵글, 라이팅, 전환)
Phase 4: 촬영 체크리스트
  └ 아이템 리스트, 사이즈, 코디 구성
```

**`shortform-planner` 공유 결정:**
폭력적인 식단, Pit Stop, Shirtless 3개 시리즈는 "소스 수집 → 리서치 → 숏폼 스크립트 → 촬영 체크리스트" 구조가 동일. 하나의 `shortform-planner` 에이전트로 통합하고, 시리즈명을 파라미터로 받아 톤/구조를 분기하는 방식이 효율적.

**복잡도:** ★★★☆☆
**특성:** 시네마틱 필름, 아티스틱, 장기 기획

Phase 1: 컨셉 기획

## 4. 에이전트 구현 우선순위

### Tier 1 — 즉시 구현 가능 (레퍼런스 충분)

| 순서 | 에이전트 | 이유 |
|---|---|---|
| 1 | `hoodies-planner` | 안성희 기획 680줄 + 마스터플랜 680줄 = 풍부한 레퍼런스. 패턴이 명확 |
| 2 | `shortform-planner` | 봤어? 스킬 + Pit Stop 호스트가이드 = 구조 참고 가능. 3시리즈 통합 |

### Tier 2 — 1~2편 기획 후 구현

| 순서 | 에이전트 | 이유 |
|---|---|---|
| 3 | `false9-planner` | 기존 레퍼런스 부족. 첫 EP 기획 과정을 기록 → 패턴 추출 |
| 4 | `hood-residents-planner` | 반구조화 특성상 패턴 정립 필요. 2~3편 기획 후 자동화 |

### Tier 3 — 전체 시스템 통합

| 순서 | 에이전트 | 이유 |
|---|---|---|
| 5 | `content-team-orchestrator` | Layer 2 에이전트가 3개 이상 완성된 후 오케스트레이터 구현 |

### 구현 마일스톤

```
Session A: hoodies-planner 에이전트 작성
  └ 마스터플랜 + 안성희 기획을 레퍼런스로 주입
  └ 5단계 파이프라인 구현
  └ 테스트: 새 출연자(고예원 or 백주현) 기획 시뮬레이션

Session B: shortform-planner 에이전트 작성
  └ 3시리즈(폭력적인 식단, Pit Stop, Shirtless) 통합
  └ 시리즈 파라미터에 따른 톤/구조 분기
  └ 테스트: 실제 장소/테마로 스크립트 생성

Session C: false9-planner + hood-residents-planner
  └ 첫 EP 기획 경험이 쌓인 후
  └ 기획 과정을 에이전트 프롬프트로 포맷팅

Session D: content-team-orchestrator
  └ 주간 플랜 입력 인터페이스
  └ TeamCreate API로 병렬 디스패치
  └ 산출물 수집 + 통합 리뷰
```

## 5. 오케스트레이터 설계

### 5-1. 역할

`content-team-orchestrator`는 수현님이 주간 콘텐츠 플랜을 넘기면:
1. 시리즈별 에이전트를 병렬로 디스패치
2. 각 에이전트의 산출물을 수집
3. 전체 일관성 리뷰
4. 통합 산출물 + 진행 현황 보고

### 5-2. 인풋 형식

```markdown
## 이번 주 콘텐츠 플랜

### Playbook
- 주제: [대주제]
- 게스트: [이름] ([배경])

### 5-3. 디스패치 로직

```
입력: 주간 플랜

1. 플랜 파싱 → 시리즈별 태스크 분리

2. 병렬 에이전트 디스패치 (TeamCreate API)
   ├─ Agent "playbook": playbook-planner에 (주제, 게스트, EP번호) 전달
   ├─ Agent "bwasseo": bwasseo-script 스킬로 (링크/주제) 전달
   ├─ Agent "shortform-food": shortform-planner에 (폭력적인 식단, 장소) 전달
   ├─ Agent "shortform-cafe": shortform-planner에 (Pit Stop, 카페) 전달
   ├─ Agent "shortform-style": shortform-planner에 (Shirtless, 테마) 전달
   └─ Agent "hoodies": hoodies-planner에 (출연자, 작업 범위) 전달

3. 산출물 수집 (각 에이전트 완료 시)
   └ 파일 경로, 주요 내용 요약

4. 통합 리뷰
   └ 시리즈 간 주제 겹침 체크
   └ 톤 일관성 확인
   └ 누락 항목 플래그

5. 산출물 리포트
   └ 이번 주 기획 완료 현황
   └ 생성된 파일 목록
   └ 리뷰 코멘트
```

### 5-4. 오케스트레이터 에이전트 구현 시 고려사항

- **TaskCreate/TaskUpdate API** 활용하여 에이전트별 진행 상태 추적
- **SendMessage** 로 에이전트 간 의존성 있는 경우 조율 (예: Hoodies 촬영 가이드가 Playbook 게스트 정보를 참조해야 하는 경우)
- **선택적 디스패치**: 주간 플랜에 포함된 시리즈만 에이전트 런칭 (매주 7개 다 하는 건 아님)
- **사용자 피드백 루프**: Hoodies 세트 구조, Playbook 소주제 등은 사용자 컨펌이 필요 → 오케스트레이터가 중간에 사용자에게 리뷰 요청

## 8. 구현 시 주의사항

### 8-1. 에이전트 크기 제한

- YAML frontmatter + system prompt = 하나의 `.md` 파일
- 시스템 프롬프트가 너무 길면 컨텍스트 소모 → 핵심만 프롬프트에, 디테일은 `references/` 폴더에 분리
- Hoodies처럼 레퍼런스가 방대한 경우: 에이전트가 런타임에 `read`로 레퍼런스 로드

### 8-2. 스킬 → 에이전트 전환 기준

현재 스킬로 존재하는 `bwasseo-script`를 에이전트로 전환할 필요가 있는지?
- **현재: 불필요.** 단일 태스크 파이프라인이라 스킬로 충분
- **미래: 전환 고려 시점** — 봤어? 에피소드를 주간 단위로 2~3개 병렬 생성해야 할 때, 오케스트레이터와 통합이 어려울 때

### 8-3. 오케스트레이터 점진적 구축

처음부터 완전한 오케스트레이터를 만들지 말 것.

```
v0.1: 2~3개 에이전트를 수동으로 병렬 디스패치 (Agent 도구 직접 호출)
v0.2: 간단한 오케스트레이터 (플랜 파싱 + 디스패치만)
v0.3: 산출물 수집 + 통합 리뷰 추가
v1.0: 완전 자동화 (플랜 입력 → 산출물 + 리포트 출력)
```

### 8-4. 테스트 전략

각 에이전트 완성 후:
1. **실제 소재로 테스트** — 가상 소재가 아닌 실제 기획 소재로 실행
2. **산출물 품질 검증** — 기존 수동 기획 결과물과 비교
3. **사용자 피드백 루프 동작 확인** — 중간 확인이 필요한 단계에서 정상 작동하는지

[월요일 오전]

     ↓ 오케스트레이터 디스패치

  Agent 1 (playbook-planner): EP.5 소주제 브레인스토밍 → 사용자 피드백 대기

  Agent 3 (shortform-planner:식단): 머슬밥 스크립트 → 완료
  Agent 4 (shortform-planner:cafe): 러너스 로스터리 스크립트 → 완료
  Agent 5 (hoodies-planner): 고예원 SET 2 촬영 가이드 → 완료

     ↓ 사용자 피드백 (Playbook 소주제 확정)

  Agent 1: 병렬 리서치 → 기획안 → 호스트 스크립트 → 완료

     ↓ 오케스트레이터 수집

[산출물 리포트]
  - 5개 시리즈 기획 완료
  - 생성 파일 7개
  - 리뷰 코멘트: "봤어? 스크립트와 Pit Stop 카페가 둘 다 러닝 앵글 → 차별화 필요"
```
